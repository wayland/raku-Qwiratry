## Work Package WP04: User Story 2 – Dependency Graph Generation (Priority: P2)

**Goal**: Auto-generate Mermaid dependency graph showing blocking relationships between features.
**Independent Test**: Dependency graph correctly shows all blocking relationships from feature metadata, graph syntax is valid Mermaid, and circular dependencies are detected.
**Prompt**: `/tasks/planned/WP04-dependency-graph-generation.md`

### Included Subtasks
- [x] T015 Build dependency graph from feature metadata (dependencies arrays)
- [x] T016 [P] Detect circular dependencies and report errors
- [x] T017 [P] Generate Mermaid flowchart syntax from dependency relationships
- [x] T018 [P] Embed Mermaid diagram in traceability map markdown
- [x] T019 Validate graph structure (no self-dependencies, valid feature references)

### Implementation Notes
- Read `dependencies` arrays from all meta.json files
- Build directed graph: Feature A --> Feature B means "A blocks B"
- Use Mermaid syntax: `graph TD\n  A[001-feature] --> B[002-feature]`
- Embed as code block with `mermaid` language tag in markdown

### Parallel Opportunities
- Circular dependency detection (T016), Mermaid generation (T017), embedding (T018), validation (T019) can proceed in parallel once graph is built (T015)

### Dependencies
- Depends on WP02 (feature metadata reading), WP03 (traceability map structure).

### Risks & Mitigations
- Circular dependencies: Detect and report clearly, prevent infinite loops
- Invalid feature references: Validate all dependencies reference existing features

## Work Package WP05: User Story 3 – Coverage Verification Script (Priority: P2)

**Goal**: Implement CLI script that verifies spec section coverage and reports results in text and JSON formats.
**Independent Test**: Script runs in under 5 seconds, correctly identifies covered/uncovered sections, reports broken links, and outputs valid JSON when requested.
**Prompt**: `/tasks/planned/WP05-coverage-verification-script.md`

### Included Subtasks
- [x] T020 Implement coverage calculation algorithm (check section against feature spec_sections)
- [x] T021 [P] Implement subsection coverage inheritance (3.2.1.1 covered if 3.2.1 covered)
- [x] T022 [P] Implement broken link detection (check feature directories exist)
- [x] T023 [P] Implement CLI argument parsing (--json, --verbose, --generate-map, --help)
- [x] T024 [P] Implement text output formatter (human-readable coverage report)
- [x] T025 [P] Implement JSON output formatter (structured data for CI/CD)
- [x] T026 Add logging (INFO for steps, ERROR for failures, WARN for uncovered sections)
- [x] T027 [P] Add file path validation to prevent directory traversal

### Implementation Notes
- Coverage calculation: For each section, check if it or any parent is in a feature's spec_sections
- CLI should support: `raku scripts/verify-spec-coverage.raku [--json] [--verbose] [--generate-map]`
- Text output: Coverage percentage, uncovered sections list, broken links, dependency graph status
- JSON output: Structured object matching contract in `contracts/coverage-script-api.md`
- Performance target: <5 seconds execution time

### Parallel Opportunities
- Coverage calculation (T020), subsection inheritance (T021), broken link detection (T022), CLI parsing (T023), text formatter (T024), JSON formatter (T025), logging (T026), path validation (T027) can all proceed in parallel once core algorithm is defined

### Dependencies
- Depends on WP02 (parsing), WP03 (mapping), WP04 (dependency graph).

### Risks & Mitigations
- Performance issues: Cache parsed Specification.md, process features in single pass
- Invalid JSON output: Validate JSON structure before output, test with jq
- Path traversal: Validate all file paths, reject paths with `..` components
